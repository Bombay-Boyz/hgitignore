# Conformance Methodology

This document defines how "matches real Git" is measured. It exists so that
"conformance" is a falsifiable, automatable claim rather than a vibe.

---

## Reference Implementation

The reference implementation is `git check-ignore`, invoked as:

```bash
git check-ignore --verbose --no-index -- <path>
```

- `--verbose` is used so the fixture-generation script can record *which*
  pattern and source file produced the result, not just the result itself —
  this gives much better triage information when a fixture's expectation
  needs investigating later, even though `hgitignore`'s own `MatchResult`
  collapses to a binary outcome (see
  [`SPEC.md`](../SPEC.md#matchresult)).
- `--no-index` is used so fixture generation does not require a populated
  Git index — fixtures are pattern-file + path combinations, not real
  repository states.
- Exit code `0` means the path is ignored; exit code `1` means it is not;
  any other exit code is a fixture-generation error, not a valid fixture.

## Supported Git Version Matrix

Tracked in [`compatibility.md`](compatibility.md). Both the **minimum
supported** and **latest stable** Git versions are run against the full
corpus in the [Phase 12](ROADMAP.md#phase-12--git-parity) pipeline — not
just one or the other — because Git's own ignore-matching behavior has
changed in minor ways across versions historically, and we want to know
which version(s) `hgitignore` is actually compatible with rather than
assuming "compatible with one version" implies "compatible with all of
them."

## Parity Methodology

A fixture's `expected` field is **never** hand-written based on what someone
believes Git does. It is always generated by:

1. Materializing the fixture's `patterns` into a real `.gitignore` file in a
   scratch directory.
2. Materializing the fixture's `path` as an actual file (or directory, for
   directory-only pattern fixtures) in that scratch directory.
3. Running `git check-ignore` (per the invocation above) against it.
4. Recording the result as `ignored` or `not-ignored` (matching the
   `expected` field's vocabulary in the fixture format below).

This is automated via `hgitignore-testkit`'s fixture-generation helpers, not
done by hand per-fixture — hand-authoring thousands of fixtures and their
expected values independently would be both impractical and a likely
source of the very errors this methodology exists to avoid.

## Fixture Format

```yaml
patterns:
  - "*.log"

path: app.log

expected: ignored
```

Fields:

- `patterns` — an ordered list of pattern lines, in the order they'd appear
  in a single `.gitignore` file. (Multi-file precedence scenarios — see
  [`semantics.md` §4](semantics.md#4-file-precedence) — use an extended
  format with a `files` list instead of a flat `patterns` list; this
  extended format is finalized as part of
  [Phase 2](ROADMAP.md#phase-2--conformance-specification) alongside the
  basic format above.)
- `path` — the candidate path being tested.
- `expected` — either `ignored` or `included`. No other values.

Optional fields, added as needed once Phase 2's bootstrap fixtures reveal
the need:

- `description` — free text explaining what edge case this fixture covers.
  Strongly encouraged for any fixture covering §3.1's directory-exclusion
  exception or any other non-obvious case.
- `git_version` — if a fixture is known to produce different results across
  Git versions, this records which version it was generated against. Absent
  for fixtures expected to be version-stable.

## Triaging Disagreements

When `hgitignore`'s output disagrees with a fixture's `expected` value,
triage in this order:

1. **Re-verify the fixture against real Git, fresh.** Fixture generation
   could have been run against a stale or wrong Git version, or the
   scratch-directory setup could have a bug. Rule this out first.
2. **Check `semantics.md` against the (re-verified) real Git behavior.** If
   `semantics.md`'s documented rule doesn't actually match what Git does,
   `semantics.md` is wrong and must be corrected — with the worked-example
   update this implies.
3. **Only then conclude the implementation is wrong**, and fix it.

Skipping straight to "fix the implementation to match the fixture" without
steps 1–2 risks silently encoding a wrong fixture or a wrong semantics.md
rule into the codebase permanently. This triage order is itself enforced by
review, not tooling — reviewers should ask "did you re-verify against real
Git?" on any PR that changes matcher behavior to fix a fixture disagreement.
